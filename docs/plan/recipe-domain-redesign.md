# 配方领域模型与冲煮记录流程改造方案

## 1. 背景

当前产品把配方库记录、AI 推荐结果和日记使用的配方混在同一套引用逻辑中，导致以下问题：

- AI 推荐只存在于页面状态，离开豆子详情页后丢失，也无法直接保存到配方库。
- 日记只保存配方快照，但“再冲一杯”和“最佳配方”仍依赖配方库 ID；原配方修改或删除后，可能无法准确复现当时参数。
- 新建日记只能选择配方，不能在保存前按本次冲煮情况调整参数。
- 配方参数大量使用字符串，缺少咖啡粉、冲煮水和冰的克重，无法可靠计算粉水比或进行结构化编辑。
- 六维评分默认值缺乏引导，用户难以理解中间分数的含义。
- 删除操作没有确认，删除豆子时还会同时删除关联日记，但界面没有说明影响范围。

本次改造将“配方内容”和“配方记录”分离，并明确 AI 推荐、配方库记录、日记配方快照三种对象的边界。

## 2. 目标

1. AI 推荐按豆子和冲煮方式持久保存，重新进入豆子详情页仍可查看。
2. AI 推荐可以一键复制到配方库，但未保存的 AI 推荐不出现在配方页。
3. 配方参数结构化，增加咖啡粉、冲煮水和冰的克重字段。
4. 新建日记时，选择配方库记录只用于快速填充，之后可以自由修改本次配方。
5. 每篇日记保存完全独立的配方快照，不受配方库后续修改或删除影响。
6. “最佳配方”改为基于实际冲煮结果的“最佳杯”，并准确复现该篇日记的参数。
7. 日记详情页支持把本次配方另存到配方库。
8. 改善六维评分的解释、分值文案和进阶维度折叠体验。
9. 为危险删除操作增加明确的影响提示和确认。

## 3. 非目标

本次不包含以下能力：

- 云同步、账号系统或跨设备同步。
- 配方版本历史和差异比较。
- 自动根据六维评分计算“最佳杯”。最佳杯始终由用户明确指定。
- 在新建日记的快速填充区展示未保存的 AI 推荐。
- 自动把日记配方保存进配方库。
- 对冲煮模拟器的数据结构和交互进行整合。

## 4. 领域对象与可见范围

### 4.1 三种配方对象

| 类型       | 存储位置                  | 配方页展示 | 新建日记快速填充               | 生命周期                                 |
| ---------- | ------------------------- | ---------- | ------------------------------ | ---------------------------------------- |
| AI 推荐    | `aiSuggestions` 表        | 否         | 否                             | 按豆子和冲煮方式持久保存，重新生成时覆盖 |
| 配方库记录 | `recipes` 表              | 是         | 是                             | 初始内置或用户明确保存                   |
| 日记配方   | `journals.recipeSnapshot` | 否         | 仅通过“再冲一杯”等明确入口复制 | 永久跟随日记，不依赖其他记录             |

配方页统一展示 `recipes` 表中的全部记录，不区分内置配方和用户后来保存的配方。新建日记的快速填充区同样展示全部配方库记录；未保存的 AI 推荐和历史日记快照不出现在该区域。

### 4.2 统一配方内容

三种配方对象共享同一个结构化值对象，但各自保存独立副本：

```ts
interface RecipeContent {
  method: BrewMethod;
  coffeeGrams: number;
  brewWaterGrams: number;
  iceGrams: number;
  grind: string;
  temperatureC: number | null;
  durationSeconds: number;
  pour: string;
  steps: PourStep[];
}

interface PourStep {
  atSeconds: number;
  targetWaterGrams: number;
  note: string;
}
```

字段约束：

- `coffeeGrams > 0`。
- `brewWaterGrams > 0`。
- `iceGrams >= 0`，热冲和冷萃通常为 `0`。
- `temperatureC` 对热冲、冰冲为有效数字；冷萃允许为 `null`，界面显示“冷水 / 室温水”。
- `durationSeconds > 0`。
- `steps[].atSeconds >= 0`，按时间升序保存。
- `steps[].targetWaterGrams` 表示累计注入的冲煮水，不包含冰。
- 最后一个注水步骤的累计水量原则上不超过 `brewWaterGrams`；不一致时给出表单提示，但允许用户确认后保存，以兼容浸泡、旁路水等特殊配方。

粉水比不再作为独立字段保存，而是通过总用水量派生：

```ts
const totalWaterGrams = brewWaterGrams + iceGrams;
const ratio = totalWaterGrams / coffeeGrams;
```

界面统一显示为 `1:${formatRatio(ratio)}`。冰冲必须同时展示“冲煮水”和“冰”，避免总水量含义不清。

### 4.3 配方库记录

```ts
interface SavedRecipe extends RecipeContent {
  id: string;
  name: string;
  createdAt: number;
  updatedAt: number;
  source?: {
    type: "manual" | "ai" | "journal";
    sourceId?: string;
  };
  needsReview?: boolean;
}
```

- 所有配方库记录地位相同，均可编辑、删除和用于快速填充。
- `source` 只用于解释来源和避免重复保存，不建立内容同步关系。
- 保存后修改源 AI 推荐或源日记，不更新已保存配方。
- 修改已保存配方，也不更新源 AI 推荐或源日记。

### 4.4 AI 推荐记录

```ts
interface AiSuggestion {
  id: string; // `${beanId}:${method}`
  beanId: string;
  method: BrewMethod;
  content: RecipeContent;
  why: string;
  source: "ai" | "local";
  generatedAt: number;
  beanFingerprint: string;
  savedRecipeId?: string;
}
```

规则：

- `(beanId, method)` 唯一，一支豆最多同时保留热冲、冰冲、冷萃各一条推荐。
- 首次生成后写入数据库，重新进入豆子详情页直接读取并展示。
- 重新生成只覆盖当前冲煮方式的推荐。
- AI 未配置或调用失败后用户选择本地规则，结果同样持久保存，`source=local`。
- `beanFingerprint` 根据名称、产区、处理法、烘焙度、烘焙日和风味生成。
- 豆子资料变化后不自动删除旧推荐；页面显示“豆子资料在本推荐生成后发生过变化”，并提供重新生成操作。
- 删除豆子时级联删除该豆子的全部 AI 推荐。

### 4.5 日记配方快照

```ts
interface RecipeSnapshot extends RecipeContent {
  name: string;
  source: {
    type: "saved-recipe" | "best-journal" | "manual" | "legacy";
    sourceId?: string;
  };
}

interface Journal {
  id: string;
  beanId: string;
  createdAt: number;
  recipeSnapshot: RecipeSnapshot;
  dims: BrewDimensions;
  notes: string;
  aiReview?: AnalysisResult;
  savedAsRecipeId?: string;
}
```

- 日记不再依赖 `recipeId` 才能展示或复现。
- `sourceId` 仅记录本次草稿最初从哪里填充，不能用于重新读取配方内容。
- 日记保存后，所有展示、分析和“再冲一杯”操作都只使用 `recipeSnapshot`。
- 删除或修改配方库记录，不影响任何历史日记。

## 5. 核心业务流程

### 5.1 AI 推荐与一键保存

豆子详情页按热冲、冰冲、冷萃分别读取推荐：

1. 当前方式没有推荐时，显示“生成推荐配方”。
2. 有推荐时，直接展示参数、步骤、推荐理由、来源和生成时间。
3. 用户点击“重新生成”，成功后覆盖当前方式的旧推荐。
4. 用户点击“一键保存到配方库”，复制推荐内容到 `recipes`。
5. 默认名称为“豆子名称 · AI 热冲/冰冲/冷萃”。
6. 保存成功后显示“已保存到配方库”，并允许跳转查看该配方。
7. 已保存配方后来被删除时，清除 AI 推荐上的 `savedRecipeId`，重新允许保存。

一键保存不打开二次编辑弹窗，确保操作确实是一键完成。用户可以保存后进入配方详情修改名称和参数。

### 5.2 新建日记

页面内部维护独立的 `recipeDraft: RecipeSnapshot`，而不是直接编辑某条配方库记录。

普通入口流程：

1. 进入页面后展示当前豆子和空白/默认配方表单。
2. 快速填充区查询并展示配方库中的全部记录。
3. 选择一条配方时，将其内容深拷贝到 `recipeDraft`。
4. 用户可修改名称、方式、咖啡粉克重、冲煮水、冰、研磨、水温、总时间、注水方式和步骤。
5. 保存日记时将当前 draft 写入 `recipeSnapshot`。

切换快速填充配方时：

- 如果当前 draft 尚未被用户修改，直接覆盖。
- 如果用户已经修改过，弹出确认：“当前修改尚未保存，切换配方将覆盖这些参数。”
- 确认后覆盖；取消后保持当前 draft。

从历史日记进入：

- “用此配方再冲一杯”直接复制该日记的 `recipeSnapshot`。
- 页面顶部提示“参数来自 MM.DD 的历史冲煮”。
- 快速填充区仍展示配方库中的全部记录。
- 用户可继续修改，本次保存后生成新的独立快照。

从最佳杯进入：

- “用最佳配方再冲一杯”复制最佳日记的 `recipeSnapshot`。
- 页面顶部提示“参数来自 MM.DD 的最佳杯”。
- 不要求该快照已经保存进配方库。

### 5.3 日记详情与另存到配方库

日记详情页在完整配方参数下提供“另存到配方库”：

1. 点击后直接复制 `recipeSnapshot` 到 `recipes`。
2. 默认沿用快照名称；若名称为空，使用“豆子名称 · MM.DD”。
3. 保存后记录 `journal.savedAsRecipeId`，按钮变为“已保存到配方库”。
4. 已保存配方被修改，不影响日记快照。
5. 已保存配方被删除时，清除 `savedAsRecipeId`，日记重新允许另存。
6. 同一篇日记默认只关联一个另存结果，避免误触产生重复配方；用户如需多个版本，可进入配方页复制或编辑。

当前产品保存日记后会进入日记详情并生成复盘，因此日记详情页即为本次冲煮的最终结果页，不新增独立结果路由。

## 6. 最佳杯逻辑

“最佳配方”改名为“最佳杯”，其依据是用户实际完成并评价过的一次冲煮，而不是配方库中的抽象参数。

```ts
interface Bean {
  // existing fields
  bestJournalId?: string;
}
```

规则：

- 只有日记可以被标记为最佳杯。
- AI 推荐不能直接成为最佳杯，必须先用于实际冲煮并生成日记。
- 配方库记录不能直接成为最佳杯。
- 每支豆最多只有一篇最佳日记；标记新的最佳杯会替换旧值。
- 在当前最佳日记上再次点击，可取消最佳杯标记。
- 删除最佳日记时，在同一事务中清除豆子的 `bestJournalId`。
- 删除非最佳日记不影响最佳杯。
- 将最佳日记另存到配方库，不改变最佳杯引用。
- 修改或删除另存出来的配方，不影响最佳杯。

豆子卡片继续显示 `BEST` 标识，但含义改为“该豆子已有最佳杯”。

豆子详情页的最佳杯区域展示：

- 日记日期。
- 配方名称和完整参数。
- 六维评分。
- 当时笔记，若有。
- “查看最佳杯记录”。
- “用最佳配方再冲一杯”。

日记详情页按钮文案改为：

- 未标记：“标为这支豆的最佳杯”。
- 已标记：“已是最佳杯 · 点击取消”。

## 7. 六维评分引导

### 7.1 分组

默认展开基础感受：

- 酸度
- 甜感
- 苦味

默认折叠“进阶感受”：

- 干净度
- 余韵
- 醇厚度

折叠标题显示当前摘要，例如：

`进阶感受 · 干净度 3 / 余韵 3 / 醇厚度 3`

### 7.2 维度说明

| 维度   | 具体说明                                       |
| ------ | ---------------------------------------------- |
| 酸度   | 感受酸质的强弱和明亮程度，不代表分数越高越好。 |
| 甜感   | 咖啡入口到回味中可感知的自然甜味强度。         |
| 苦味   | 烘焙和萃取带来的苦感强度，不等同于醇厚度。     |
| 干净度 | 风味是否清晰、分明，是否有浑浊或杂味。         |
| 余韵   | 咽下后风味在口中停留的时长和清晰度。           |
| 醇厚度 | 咖啡液在口中的重量、黏稠度和包裹感。           |

### 7.3 五级文案

| 维度   | 1    | 2    | 3    | 4    | 5    |
| ------ | ---- | ---- | ---- | ---- | ---- |
| 酸度   | 平淡 | 柔和 | 平衡 | 明亮 | 尖锐 |
| 甜感   | 无甜 | 微甜 | 清晰 | 甜润 | 浓郁 |
| 苦味   | 无苦 | 轻微 | 适中 | 明显 | 强烈 |
| 干净度 | 浑浊 | 略杂 | 尚可 | 清晰 | 通透 |
| 余韵   | 短促 | 较短 | 适中 | 持久 | 绵长 |
| 醇厚度 | 水感 | 轻盈 | 中等 | 饱满 | 厚重 |

每个滑块旁只常驻展示当前分数对应的词，滑动时即时更新。例如：`4/5 · 明亮`。不同时展开五个解释，避免移动端信息过载。

### 7.4 未主动评价的处理

评分初始仍为 3，但在页面状态中记录每个维度是否被用户操作过：

```ts
type TouchedDimensions = Partial<Record<keyof BrewDimensions, true>>;
```

保存时若存在未操作维度，显示一次确认：

> 还有 3 个维度尚未评价，是否按 3 分（中等）记录？

操作：

- “继续填写”：关闭确认并展开包含未评价项的分组。
- “按 3 分保存”：按当前值保存日记。

`touched` 只用于当前编辑过程，不写入日记数据库。

## 8. 删除与引用清理

### 8.1 删除豆子

入口仍位于编辑豆子页，不在豆子详情页新增删除按钮。

确认文案必须显示实际影响数量：

> 删除“豆子名称”？这会同时永久删除 4 篇冲煮日记和 3 条 AI 推荐，且无法撤销。

确认后在同一事务中删除：

- 豆子。
- 该豆子的全部日记。
- 该豆子的全部 AI 推荐。

配方库记录不随豆子删除，因为用户配方不归属于单一豆子；其来源信息可以保留失效的 `sourceId`，但界面不得依赖该引用。

### 8.2 删除日记

- 普通确认：“删除这篇日记？此操作无法撤销。”
- 如果是最佳杯，追加说明“删除后将同时清除这支豆的最佳杯标记”。
- 如果日记曾另存到配方库，不删除对应配方。

### 8.3 删除配方

- 所有配方库记录均可删除。
- 删除前普通确认。
- 不删除使用过该配方的历史日记，因为日记已有独立快照。
- 清理所有指向该配方的 `AiSuggestion.savedRecipeId` 和 `Journal.savedAsRecipeId`。
- 不影响最佳杯。

## 9. 示例数据调整

新安装时，豆子示例数据名称增加“示例 ”前缀：

- `示例 耶加雪菲`
- `示例 慧兰`
- `示例 瑰夏`

关联日记通过豆子名称自然显示示例身份，不单独修改日记标题。

初始内置配方不增加“示例”前缀。它们是可供新手参考的真实推荐配方，与其他配方统一显示在配方库中。

升级已有数据库时，只对 ID 和原始名称均匹配内置数据的豆子补充前缀。如果用户已经修改过名称，不覆盖用户内容。

## 10. 数据库升级与迁移

### 10.1 Dexie 版本

数据库最终升级到版本 4，移除配方分类字段和索引：

```ts
this.version(4).stores({
  beans: "id, no, name, bestJournalId",
  recipes: "id, method, updatedAt",
  journals: "id, beanId, createdAt, savedAsRecipeId",
  aiSuggestions: "id, beanId, [beanId+method], generatedAt, savedRecipeId",
  settings: "id",
  meta: "id",
});
```

### 10.2 旧配方字段转换

旧字段到新字段的通用转换：

- `ratio: "1:16"` 解析出比例值 `16`。
- `temp: "92°C"` 解析为 `temperatureC=92`；无法解析时设为 `null` 并标记 `needsReview=true`。
- `time: "2:30"` 解析为 `durationSeconds=150`。
- `steps[].t` 转换为 `atSeconds`。
- `steps[].water` 提取箭头后的累计克重作为 `targetWaterGrams`。
- 若能取得最后一步累计水量，则把它作为 `brewWaterGrams`，并通过比例反推 `coffeeGrams`。
- 若不能取得累计水量，则默认 `coffeeGrams=15`，按旧比例计算 `brewWaterGrams`，并标记 `needsReview=true`。
- 普通旧配方默认 `iceGrams=0`；无法可靠拆分水和冰的冰冲配方标记 `needsReview=true`。

内置配方使用明确迁移值，避免通用解析产生歧义：

| ID     | 咖啡粉 | 冲煮水 |  冰 |
| ------ | -----: | -----: | --: |
| `v60`  |  16.7g |   250g |  0g |
| `one`  |  15.6g |   250g |  0g |
| `ice`  |    15g |   160g | 65g |
| `mine` |    15g |   240g |  0g |

旧日记的 `recipeSnapshot` 使用同一转换函数独立迁移，不能在迁移后重新引用配方库。

### 10.3 最佳配方迁移

旧数据中的 `Bean.bestRecipeId` 转换为 `bestJournalId`：

1. 查询该豆子下 `recipeId === bestRecipeId` 的日记。
2. 如果有多篇，选择 `createdAt` 最新的一篇。
3. 将其 ID 写入 `bestJournalId`。
4. 如果没有匹配日记，清除旧最佳引用，不自动选择其他日记。

完成迁移后，业务代码不再读取 `bestRecipeId` 或依赖 `Journal.recipeId`。

## 11. 页面改动清单

### 豆子列表

- 示例豆名称增加前缀。
- `BEST` 表示存在有效的 `bestJournalId`。

### 豆子详情

- AI 推荐按方式持久展示。
- 增加重新生成、一键保存和保存状态。
- 豆子资料变化时显示推荐已过期提示。

## 12. 错误处理与边界情况

- AI 生成失败：保留旧推荐，不用失败结果覆盖；提供重试和本地规则。
- AI 返回非法克重或时间：拒绝写入并显示可理解的结构校验错误。
- 配方库为空：新建日记仍允许手动填写，不阻塞记录。
- 配方库为空：快速填充区显示空状态，但完整配方表单保持可用。
- 快速填充来源被其他页面删除：当前 draft 已是副本，可以正常保存。
- 最佳日记不存在：读取豆子详情时清理悬空 `bestJournalId`，不静默选择其他日记。
- 删除已另存的配方：日记内容不变，并恢复“另存到配方库”按钮。
- 重复点击保存：保存按钮在事务完成前禁用，避免重复日记或重复配方。
- 数据库迁移字段无法解析：保留尽可能多的原始含义，设置 `needsReview`，在编辑配方时提示用户确认克重和时间。
